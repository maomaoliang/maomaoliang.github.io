MCP 协议原理学习

MCP 是什么

MCP（Model Context Protocol）= 一套”AI 调用外部工具”的标准接口规范。就像 USB 是”电脑连外设”的标准，MCP 是”AI 连工具”的标准。

三个角色各做什么

Claude（AI 大脑）

  • 职责：理解人话 + 做决策
  • 收到用户的话 → 判断该调哪个工具 → 构造参数 → 收到结果后翻译成自然语言回复
  • Claude 不会自己去操作文件，它只负责”想”和”说”

MCP 协议（通信规矩）

  • 职责：规定消息格式，让 AI 和工具能互相听懂
  • 协议本身不干活，它只是一张规范表，规定了”谁说什么、怎么说、用什么格式说”
  • 核心格式：JSON-RPC

我写的函数（干活的手脚）

  • 职责：真正执行操作（读写文件、调 API 等）
  • 不懂人话，也不知道 Claude 的存在，就是接到参数、干活、返回结果

类比

角色 类比
Claude 老板（决定做什么）
MCP 协议 工单系统（规定怎么下单、怎么交付）
我写的函数 工人（按工单干活）

为什么需要协议

理解力 ≠ 连接能力。 协议解决的不是”懂不懂”的问题，是”通不通”的问题。

Claude 和我的函数是两个完全独立的进程，甚至不在同一台电脑上。Claude 再聪明，也摸不到我的函数。需要协议建立通道 + 统一格式。

为什么选 stdio 而不是 HTTP

  • 不需要端口和网络
  • 纯本地，父子进程才能通信，安全性高
  • 启动即可用，无需额外配置

FastMCP 框架的作用

FastMCP 是”翻译官”，让我不用手写 JSON-RPC：

  • 我只需要写普通 Python 函数 + @mcp.tool() 装饰器
  • FastMCP 自动把函数签名和 docstring 转成工具描述
  • 自动处理 JSON-RPC 的序列化/反序列化

关键收获

  1. MCP 协议是连接标准，不是 AI 能力的一部分
  2. 大模型的理解力不等于连接能力，需要协议建立通道
  3. 写 MCP 工具很简单：普通函数 + 装饰器 + 好的 docstring
  4. 协议细节被 FastMCP 封装，开发者只需关注业务逻辑
